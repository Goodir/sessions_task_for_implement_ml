## Концепты MLOps

### RabbitMQ

RabbitMQ в проекте не реализовывался, потому что для текущего API это было бы лишнее усложнение. Сейчас `/predict` работает синхронно: клиент отправляет признаки, Flask-сервис сразу делает предсказание через ONNX-модель и возвращает результат

В более крупной системе RabbitMQ можно было бы добавить для асинхронных задач. Например, для batch-предсказаний, фоновой обработки логов или интеграции с внешними сервисами:

```text
Flask API -> RabbitMQ -> worker
```

Тогда основной API не ждал бы тяжелую фоновую обработку, а просто отправлял бы событие в очередь. Worker забирал бы сообщения из очереди и отдельно сохранял результат, лог или метрики.

Такой подход также упрощает масштабирование. Если нагрузка растет, можно запускать несколько worker'ов, которые будут параллельно обрабатывать сообщения из одной очереди. При этом API остается легковесным и отвечает только за прием запросов

### Логирование

В проекте добавлено JSON-логирование запросов к `/predict`. В лог пишутся `timestamp`, `request_id`, endpoint, статус ответа, версия модели, количество строк, предсказание и время обработки.

Полные признаки клиента я не логирую, потому что в задаче кредитного скоринга это могут быть чувствительные данные.

Сейчас логи пишутся в stdout контейнера. Их можно посмотреть через:

```bash
docker compose logs ml-api
```

Если нужны только структурированные логи приложения, можно отфильтровать их по префиксу:

```bash
docker compose logs ml-api | grep APP_LOG
```

В production такие логи можно было бы собирать в централизованную систему логирования, например в ELK-стек. В такой схеме Logstash собирает и обрабатывает логи разных сервисов, Elasticsearch хранит их и позволяет по ним искать, а Kibana используется для визуализации


### MLflow

MLflow в проекте не подключался, но он хорошо подходит для следующего этапа. Его можно использовать, чтобы сохранять эксперименты по обучению модели

Например, в MLflow можно логировать:

```text
тип модели
параметры модели
F1-score
Precision
Recall
путь к сохраненной модели
версию модели
```

Это удобно, если сравнивать несколько моделей, например LogisticRegression, RandomForestClassifier и GradientBoostingClassifier. Тогда выбор модели будет не по памяти или ручным заметкам, а по сохраненным результатам экспериментов

Еще MLflow можно использовать как model registry. Тогда версии `v1` и `v2` можно было бы хранить и переключать более системно

### Бизнес-метрики

Кроме F1 и Precision, для банка важны метрики, которые ближе к деньгам и риску

Первая метрика - ожидаемые потери по клиенту:

```text
expected_loss = probability_of_default * LIMIT_BAL * loss_given_default
```

`probability_of_default` дает модель, `LIMIT_BAL` есть в датасете, а `loss_given_default` задается как бизнес-допущение. Например, какую долю лимита банк в среднем теряет, если клиент уходит в дефолт

Вторая метрика - доля одобренных клиентов при заданном уровне риска:

```text
approval_rate = approved_clients / all_clients
```

Например, можно одобрять клиентов, у которых вероятность дефолта ниже выбранного порога:

```text
approved = probability_of_default < threshold
```

Тут идея не в том, чтобы просто отклонять больше клиентов. Хорошая модель должна помогать находить больше нормальных клиентов, но при этом держать ожидаемые потери на приемлемом уровне
