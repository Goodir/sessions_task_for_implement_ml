# A/B-тестирование моделей

Этот раздел описывает, как можно организовать A/B-тест для сравнения двух версий модели дефолта:

* `v1` - текущая модель
* `v2` - новая версия модели

Идея теста: часть заявок обрабатывается старой моделью, часть - новой. После накопления нужного объема данных сравниваем качество моделей и бизнес-метрики

## Постановка A/B-теста

### Группы

Контрольная группа:

```text
model_version = v1
```

Тестовая группа:

```text
model_version = v2
```

Одна заявка или один клиент должны попадать только в одну группу. Это нужно, чтобы не смешивать результаты разных моделей

### Разделение трафика

Базовый вариант - случайное распределение 50/50:

```text
50% заявок -> model_v1
50% заявок -> model_v2
```

В реальном сервисе лучше делать разбиение стабильно по `client_id`, а не случайно на каждый запрос:

```text
hash(client_id) % 2 == 0 -> v1
hash(client_id) % 2 == 1 -> v2
```

Так один и тот же клиент не будет прыгать между версиями модели

## Размер выборки и длительность теста

Длительность теста лучше считать не вручную "на глаз", а от нужного размера выборки

Для этого сначала задаются параметры теста:

```text
alpha = 0.05
power = 0.8
MDE = минимальный эффект, который мы хотим уметь заметить
baseline_metric = текущее значение метрики у model_v1
daily_requests = среднее количество заявок в день
traffic_share = доля трафика в тесте
```

Например, если основной метрикой является Recall по классу дефолта, то сначала нужно понять текущий Recall у `v1`, потом задать MDE. Например, хотим заметить рост Recall хотя бы на 5%

Дальше рассчитывается нужное количество наблюдений в каждой группе. После этого длительность можно оценить так:

```text
test_duration_days = sample_size_per_group * 2 / daily_requests
```

Если в тест включается не весь трафик, а только часть, формула становится такой:

```text
test_duration_days = sample_size_per_group * 2 / (daily_requests * traffic_share)
```

На практике я бы еще добавил минимальный срок теста, например 2 недели. Это нужно, чтобы не поймать случайную дневную просадку или перекос по дням недели.

## Метрики

### Основная техническая метрика

Основная метрика - `F1-score` для класса дефолта

Она подходит, потому что в задаче дефолта важны и Precision, и Recall. Нужно находить рискованных клиентов, но при этом не помечать слишком много нормальных клиентов как дефолтных

```text
F1 = 2 * Precision * Recall / (Precision + Recall)
```

### Дополнительная техническая метрика

Дополнительно я бы смотрел `Recall` по классу дефолта:

```text
Recall = найденные дефолты / все реальные дефолты
```

Для банка это важная метрика, потому что пропущенный дефолтный клиент может привести к финансовым потерям. Но Recall нельзя смотреть отдельно от Precision, иначе модель может начать слишком часто предсказывать дефолт

### Бизнес-метрики

Первая бизнес-метрика - ожидаемые потери:

```text
expected_loss = probability_of_default * LIMIT_BAL * loss_given_default
```

`probability_of_default` дает модель, `LIMIT_BAL` есть в датасете, а `loss_given_default` задается как бизнес-допущение. Например, это может быть доля лимита, которую банк в среднем теряет при дефолте

Вторая бизнес-метрика - доля одобренных клиентов при заданном уровне риска:

```text
approval_rate = approved_clients / all_clients
```

Например, можно считать клиента одобренным, если вероятность дефолта ниже выбранного порога:

```text
approved = probability_of_default < threshold
```

Тут задача не в том, чтобы просто отклонять больше клиентов. Хорошая модель должна помогать находить больше надежных клиентов при приемлемом уровне риска

## Статистический анализ

После завершения теста я сравниваю не просто точечные значения метрик, а разницу между группами `v1` и `v2`

Основная логика такая:

```text
diff = metric_v2 - metric_v1
```

Дальше нужно понять, эта разница похожа на реальный эффект или могла появиться случайно из-за выборки

### Гипотезы

Для основной метрики `F1-score`:

```text
H0: F1_v2 <= F1_v1
H1: F1_v2 > F1_v1
```

То есть нулевая гипотеза говорит, что новая модель не лучше текущей. Альтернативная - что новая модель лучше по F1

Для дополнительной метрики `Recall` по классу дефолта:

```text
H0: Recall_v2 <= Recall_v1
H1: Recall_v2 > Recall_v1
```

Recall здесь важен, потому что банк больше рискует, если модель пропустит клиента, который потом уйдет в дефолт

Для бизнес-метрики `approval_rate` можно проверять так:

```text
H0: approval_rate_v2 = approval_rate_v1
H1: approval_rate_v2 != approval_rate_v1
```

А для `expected_loss`:

```text
H0: mean_expected_loss_v2 >= mean_expected_loss_v1
H1: mean_expected_loss_v2 < mean_expected_loss_v1
```

Тут хорошая ситуация - когда ожидаемые потери у новой модели ниже

### Какой тест использовать

Для `Recall`, `Precision` и `approval_rate` подходит z-test для пропорций, потому что эти метрики по смыслу являются долями

Например, Recall считается как доля найденных дефолтов среди всех реальных дефолтов:

```text
Recall = TP / (TP + FN)
```

Для `approval_rate` логика такая же:

```text
approval_rate = approved_clients / all_clients
```

Для `F1-score` обычный z-test напрямую не очень удобен, потому что F1 - это составная метрика из Precision и Recall:

```text
F1 = 2 * Precision * Recall / (Precision + Recall)
```

Поэтому для F1 я бы использовал bootstrap. Берем много случайных перевыборок из групп `v1` и `v2`, каждый раз считаем F1 и разницу:

```text
diff_f1 = F1_v2 - F1_v1
```

После этого по распределению `diff_f1` строим доверительный интервал

Для `expected_loss` это уже числовая метрика по клиентам:

```text
expected_loss = probability_of_default * LIMIT_BAL * loss_given_default
```

Если распределение expected loss не слишком скошенное, можно использовать t-test для сравнения средних. Если распределение сильно скошено, лучше дополнительно проверить результат непараметрическим тестом Манна-Уитни

### Доверительные интервалы

Для метрик-долей, например Recall или approval_rate, доверительный интервал строится для разницы долей:

```text
diff = p_v2 - p_v1
```

Интервал показывает, где примерно может находиться реальная разница между моделями

Если 95% доверительный интервал полностью выше 0, значит `v2` лучше по этой метрике

```text
CI = [0.01, 0.05]
```

Если интервал содержит 0, то уверенно говорить о преимуществе нельзя

```text
CI = [-0.01, 0.04]
```

Для F1 доверительный интервал строится через bootstrap. То есть мы много раз пересчитываем F1 на случайных перевыборках и берем границы 2.5% и 97.5% по получившимся значениям разницы

Для expected loss доверительный интервал строится для разницы средних:

```text
diff = mean_expected_loss_v2 - mean_expected_loss_v1
```

Тут хорошая ситуация - когда интервал полностью ниже 0, потому что это значит, что новая модель снижает ожидаемые потери

### Критерий успеха

Модель `v2` можно считать успешной, если выполняются условия:

* F1-score по классу дефолта выше, чем у `v1`, и доверительный интервал разницы не противоречит этому
* Recall по классу дефолта выше или хотя бы не хуже
* Precision не просел слишком сильно, иначе модель может начать слишком часто помечать нормальных клиентов как дефолтных
* expected loss ниже или хотя бы не растет
* approval rate не падает при том же уровне риска
* тест набрал нужный размер выборки, рассчитанный заранее через MDE, alpha и power

Если новая модель немного улучшает Recall, но сильно ухудшает Precision или бизнес-метрики, я бы не считал такой тест успешным. Для банка важно не просто найти больше дефолтов, а сохранить нормальный баланс между риском и количеством одобренных клиентов


## Связь с архитектурой API

В сервисе уже добавлен базовый A/B-интерфейс для работы с двумя версиями модели

Сейчас в `model_handler.py` загружаются две ONNX-модели:

```text
model_v1.onnx
model_v2.onnx
```

Версию модели можно выбрать двумя способами

### 1. Явно передать `model_version`

В запросе можно указать нужную версию модели:

```json
{
  "model_version": "v1",
  "features": [[...]]
}
```

Тогда API использует именно эту модель и вернет ее в ответе:

```json
{
  "model_version": "v1",
  "prediction": [1],
  "probability": [{"0": 0.33, "1": 0.67}],
  "request_id": "..."
}
```

Пример curl-запроса для `v1`:

```bash
curl -X POST "http://127.0.0.1:8080/predict" -H "Content-Type: application/json" --data-raw '{"model_version": "v1", "features": [[1, 20000, 2, 2, 1, 24, 2, 2, -1, -1, -2, -2, 3913, 3102, 689, 0, 0, 0, 0, 689, 0, 0, 0, 0]]}'
```

Пример curl-запроса для `v2`:

```bash
curl -X POST "http://127.0.0.1:8080/predict" -H "Content-Type: application/json" --data-raw '{"model_version": "v2", "features": [[1, 20000, 2, 2, 1, 24, 2, 2, -1, -1, -2, -2, 3913, 3102, 689, 0, 0, 0, 0, 689, 0, 0, 0, 0]]}'
```

### 2. Автоматически выбрать модель через `client_id`

Если `model_version` не передан, сервис смотрит на `client_id`. Дальше он считает hash от `client_id` и распределяет клиента в один из двух потоков:

```text
hash(client_id) % 2 == 0 -> v1
hash(client_id) % 2 == 1 -> v2
```

Так один и тот же клиент стабильно попадает в одну и ту же группу. Это лучше, чем случайно выбирать модель на каждый запрос

Пример запроса с автоматическим выбором модели:

```bash
curl -X POST "http://127.0.0.1:8080/predict" -H "Content-Type: application/json" --data-raw '{"client_id": "client_1", "features": [[1, 20000, 2, 2, 1, 24, 2, 2, -1, -1, -2, -2, 3913, 3102, 689, 0, 0, 0, 0, 689, 0, 0, 0, 0]]}'
```

Еще один пример с другим `client_id`:

```bash
curl -X POST "http://127.0.0.1:8080/predict" -H "Content-Type: application/json" --data-raw '{"client_id": "client_01", "features": [[1, 20000, 2, 2, 1, 24, 2, 2, -1, -1, -2, -2, 3913, 3102, 689, 0, 0, 0, 0, 689, 0, 0, 0, 0]]}'
```

В ответе будет видно, какая модель реально использовалась:

```json
{
  "model_version": "v1",
  "prediction": [1],
  "probability": [{"0": 0.33, "1": 0.67}],
  "request_id": "..."
}
```

или:

```json
{
  "model_version": "v2",
  "prediction": [1],
  "probability": [{"0": 0.33, "1": 0.67}],
  "request_id": "..."
}
```

Так можно связать логи, предсказания и результаты A/B-теста. В логах тоже сохраняется `model_version`, поэтому потом можно отдельно анализировать результаты для `v1` и `v2`
